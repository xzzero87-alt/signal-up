## 1. Problem Statement

암호화폐 시장은 24/7로 움직이고 변동성이 크다. 수동 모니터링은 핵심 진입/이탈 타이밍을 놓치게 만들거나 차트 앞에 묶이게 한다. 시판되는 자동 시그널 서비스는 (1) 월 구독료 부담, (2) 전략 블랙박스, (3) KRW 마켓 커버리지 부족이라는 한계가 있다. 본 프로젝트는 **검증 가능한 수치 기반 전략(BB + CCI)** 으로 1시간봉 KRW 마켓 시그널을 자동 감지해 텔레그램으로 알림하고, 동일 전략을 백테스트로 사전 검증하는 도구를 만든다.

**v2.0 확장:** 결과물을 **GitHub에 공개해 다른 트레이더도 자기 PC에 설치해 사용**할 수 있게 한다. 각 사용자는 자기 텔레그램 봇 토큰과 화이트리스트를 갖고 운영한다. 클라우드 호스팅·회원가입·결제·중앙 DB는 제외(=별도 SaaS 프로젝트가 필요한 영역).

## 2. Goals

| # | Goal | 측정 방법 | 목표값 |
|---|------|----------|--------|
| G1 | 시그널 누락 없이 정시 송출 | 1시간봉 마감 후 90초 이내 텔레그램 도착 비율 | ≥ 99% |
| G2 | 백테스트 신뢰도 확보 | 워크포워드 검증 구간(out-of-sample) MDD | ≤ 25% |

| G5 | 의사결정 지원 정보 노출 | 알림 메시지에 BB 위치·CCI 값·거래량 비율 동봉 | 100% |
| **G6** | **신규 사용자 온보딩 친화도** *(v2.0)* | README 따라 첫 시그널 수신까지 소요 시간(개발 경험 1년 이상 가정) | ≤ 30분 |
| **G7** | **GUI 사용성** *(v2.0)* | 백테스트 실행·시그널 확인·설정 변경을 GUI 단독으로 완료 가능 | 100% (CLI 미사용) |

## 4. User Stories

페르소나 2종으로 확장:
- **운영자 본인 (Primary)** — 직접 개발·운영하는 사용자
- **자가설치 사용자 (Secondary, v2.0)** — GitHub에서 클론·설치해 자기 PC에서 운영하는 일반 트레이더(개발 경험 1년 이상 가정)

### 4.1 시그널 수신 시점 (Primary + Secondary 공통)

- As a 트레이더, **모니터링 의무에서 벗어나** BB+CCI 진입 조건을 자동 감지받고 싶다.
- As a 트레이더, **알림 메시지만으로 즉시 판단**할 수 있도록 가격·BB 위치·CCI 값·거래량 비율을 한눈에 보고 싶다.
- As a 트레이더, **차트 이미지가 첨부**되어 시각적 맥락을 보고 싶다.
- As a 트레이더, **같은 코인의 같은 방향이 짧은 시간에 반복 알림 오는 일을 피하고 싶다**.

### 4.2 백테스트 시점

- As a 트레이더, **임계값(CCI ±100/±200, 거래량 1.0x/1.5x)을 바꿔가며 백테스트를 돌리고 싶다**.
- As a 트레이더, **워크포워드 분할 결과**로 과적합 여부를 판단하고 싶다.
- As a 트레이더, **결과를 한 페이지 HTML 리포트**로 받아 거래 목록·누적수익·MDD를 한눈에 검토하고 싶다.

### 4.3 운영 시점

- As a 운영자, **CLI 한 줄로 백그라운드 실행**할 수 있어야 한다.
- As a 운영자, **dry-run 모드**로 텔레그램 송출 없이 시그널 로직만 검증하고 싶다.
- As a 운영자, **로그를 통해 사이클별 동작과 실패 원인을 추적**할 수 있어야 한다.

### 4.4 GUI 시점 (v2.0 신설)

- As a 자가설치 사용자, **README에 적힌 한 줄 명령(`uv run signal serve`)으로 GUI를 띄우고** localhost로 접속해 시작하고 싶다 — 터미널 친숙도가 낮아도 운영 가능하도록.
- As a 자가설치 사용자, **설정 페이지에서 화이트리스트·BB/CCI 임계값·텔레그램 봇 토큰을 입력·저장**하고 즉시 반영되길 원한다 — 매번 `.env`를 직접 편집하지 않기 위해.
- As a 자가설치 사용자, **시그널 대시보드에서 최근 시그널 리스트와 코인별 상태(BB 위치, CCI, 다음 평가 시각)** 를 확인하고 싶다.
- As a 자가설치 사용자, **백테스트 페이지에서 코인·기간·모드·임계값을 입력하고 실행 버튼을 누르면 진행 상황과 결과 차트가 페이지에 표시**되길 원한다.
- As a 자가설치 사용자, **GUI에서 데몬 시작/종료 버튼**으로 라이브 루프를 제어하고 싶다 — 잘못된 설정으로 잘못된 알림을 보내지 않도록.
- As a 자가설치 사용자, **민감 정보(텔레그램 토큰)는 입력 후 마스킹되어 표시**되길 원한다 — 어깨너머 노출을 막기 위해.

## 5. Requirements

각 요구사항은 P0(필수) / P1(있으면 좋음) / P2(미래 고려) 로 분류. P0는 빠지면 시그널 시스템 자체가 성립하지 않는다.

### 5.1 Must-Have (P0)

#### 5.1.1 시그널 코어 (v1 그대로)

- **R-P0-1** — 업비트 캔들 자동 수집 (1시간봉, 200봉)
- **R-P0-2** — BB·CCI 지표 정확 계산 (외부 레퍼런스 오차 ≤ 0.01%)
- **R-P0-3** — 모드 A 시그널 평가 (평균회귀)
- **R-P0-4** — 모드 B 시그널 평가 (스퀴즈 돌파)
- **R-P0-5** — 텔레그램 알림 송출 (텍스트 + 차트 PNG)
- **R-P0-6** — 쿨다운 (코인×모드×방향 / 2시간 / 파일 영속)
- **R-P0-7** — 백테스트 엔진
- **R-P0-8** — CLI (`run`, `backtest`, `scan-once`, `doctor`, `fetch-candles`)
- **R-P0-9** — 구조화 로그 + 상관 키

> 위 9개의 수용 기준은 v1.0 PRD §5.1과 동일. 본 v2.0에서는 변경 없음.

#### 5.1.2 GUI 계층 (v2.0 신설)

##### R-P0-10 — 로컬 웹 서버 기동 (`signal serve`)
- Given 사용자가 `.env`에 `TELEGRAM_BOT_TOKEN`을 설정하고
- When `uv run signal serve`를 실행하면
- Then `http://localhost:{WEB_PORT}` 에서 대시보드가 응답하고, 백그라운드 시그널 루프(`runner`)가 함께 동작한다(asyncio TaskGroup)
- 기본 포트 8765 (충돌 시 환경변수로 변경 가능)
- 기본 바인드 주소 `127.0.0.1`만 (외부 노출 방지)

##### R-P0-11 — 설정 페이지
- 화면에서 변경 가능한 항목: 화이트리스트(콤마 분리), BB period/std_mult, CCI period, CCI 임계값(Normal/Strong), 거래량 비율(A·B), 스퀴즈 lookback/quantile, 쿨다운 시간, 텔레그램 봇 토큰 / 채팅 ID, dry-run 토글
- 저장 시 `state/settings.json`에 영속 (DESIGN.md 결정)
- 텔레그램 토큰은 입력 후 `••••••••XXXX` 형태로 마스킹 표시
- "설정 저장 후 즉시 반영" — 다음 사이클부터 새 설정 적용
- "설정 검증" 버튼 → `signal doctor` 동등 로직 실행

##### R-P0-12 — 시그널 대시보드
- 카드 영역: 데몬 상태(Running/Stopped), 다음 평가 시각, 최근 1시간 처리 코인 수, 최근 1시간 송출 시그널 수
- 표 영역: 화이트리스트 코인별 현재 상태 — 가격, BB 위치(%B), CCI, 거래량 비율, 마지막 시그널 시각·방향
- 리스트 영역: 최근 50개 시그널 (시각, 코인, 모드, 방향, 강도, 가격, 텔레그램 송출 성공 여부)
- 자동 새로고침: 30초마다 또는 봉 마감 직후 (폴링)

##### R-P0-13 — 백테스트 페이지
- 입력 폼: 코인(드롭다운), 기간(from/to), 모드(A/B/둘 다), 임계값(기본값 미리 채움)
- "실행" 버튼 → 비동기 백그라운드 잡으로 실행. 진행률 표시
- 결과 섹션: 요약 카드, 누적수익률 라인 차트, 거래 목록 테이블
- "HTML 리포트로 저장" 버튼
- 동시 실행 1건 제한

##### R-P0-14 — 데몬 제어
- "Start"/"Stop" 토글 버튼. Stop은 진행 중 사이클 완료 후 종료(graceful)
- 상태는 GUI에 실시간 반영 + 로그 기록

##### R-P0-15 — 보안 기본선
- 바인드 주소 기본 `127.0.0.1`
- 텔레그램 토큰 응답 마스킹
- `WEB_BIND≠127.0.0.1` + `WEB_AUTH_PASSWORD` 미설정 → 시작 거부
- README에 보안 경고 명시

## 6. Success Metrics

### 6.1 Leading Indicators (1주~1개월)

| 지표 | 목표 | 측정 |
|------|------|------|
| 알림 정시 도달률 | ≥ 99% | 로그 타임스탬프 차이 |
| API 에러율 | ≤ 1% | 사이클당 실패 코인 비율 |
| 시그널 송출 빈도 | 코인당 주 1~5회 | `state/signals.jsonl` |
| dry-run 평가 정확성 | 100% | 합성 캔들 단위 테스트 |
| **(v2.0) GUI 페이지 로딩** | **≤ 1s** | 로컬 5회 평균 |
| **(v2.0) 설정 저장 반영 시간** | **다음 사이클(≤ 1h)** | 설정 변경 → 첫 시그널 |

### 6.2 Lagging Indicators (1~3개월)

| 지표 | 목표 | 측정 |
|------|------|------|
| 라이브 시그널 적중률 | ≥ 50% | 시그널 후 24봉 내 BB 중심선 도달 |
| 라이브 MDD (가상 시뮬) | ≤ 25% | 라이브 시그널을 백테스트 엔진에 입력 |
| 무인 운영 7일 무중단 | ≥ 95% | systemd/cron 로그 |
| **(v2.0) 신규 설치 → 첫 시그널까지** | **≤ 30분** | 본인 또는 베타 테스터 측정 |
| **(v2.0) GitHub Star / Fork** | 참고 지표만 | - |

### 6.3 평가 시점

- **2주차** — Leading 지표만
- **1개월차** — 시그널 빈도, 백테스트 vs 라이브 일치도, GUI 사용 만족도
- **3개월차** — 적중률, MDD, 운영 안정성, 신규 설치 친화도, 자가 만족도
